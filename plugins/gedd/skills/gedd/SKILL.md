# GEDD

Use this skill for Grounded Eval-Driven Development work.

## Default entry point

Prefer the website as the first user path unless the user explicitly asks for CLI-only work.

```bash
cd grounded-evals
pip install -e ".[dev]"
grounded-evals serve
```

The website opens at `http://localhost:8080`. Local runs default to guest mode unless `ADMIN_PASSWORD` or Cognito environment variables are set.

## Web workflow

1. Open the website.
2. Start with Demos if the user is new; the app includes 17 preloaded domain scenarios.
3. Use Coach to define the agent, system prompt, runtime, and golden queries.
4. Use Eval Harness to generate responses from supported Bedrock or Anthropic models.
5. Use Tag for open coding: correct, partial, incorrect, error codes, severity, confidence, and memos.
6. Use Root Causes for axial coding with the paradigm model.
7. Use Build Judge for rubric dimensions, hard-fail rules, and judge calibration.
8. Use Report or Session Handoff to export `session.json` and supporting artifacts.

## CLI workflow

Use the CLI when the user wants automation, repeatability, or ML engineering handoff.

```bash
cd grounded-evals
grounded-evals chat --session session.json
grounded-evals validate-session --session session.json
grounded-evals handoff --session session.json
grounded-evals export --session session.json --format jsonl --output golden.jsonl
grounded-evals judge --session session.json --output judge.md
grounded-evals mlflow --session session.json --tracking-uri $ARN --run-eval
```

## Method rules

- Keep the domain expert in the problem space before writing rubrics.
- Do not start with generic failure categories such as hallucination, helpfulness, or tone.
- Generate golden queries across happy path, edge cases, adversarial inputs, ambiguity, multi-turn behavior, and error recovery.
- Name failures in the expert's own vocabulary, for example `dosage_unit_confusion` or `coverage_hallucination`.
- Treat memos as judge training data: capture why the failure matters and who is harmed.
- Validate handoff readiness before telling an ML engineer to build CI gates.

## Expected artifacts

- `session.json`: canonical handoff with agent spec, prompt, golden queries, annotations, prompt variants, and messages.
- `golden.jsonl`: exported evaluation dataset.
- `judge.md`: G-Eval judge prompt grounded in observed failures.
- MLflow experiment: generated by `grounded-evals mlflow` for SageMaker tracking and CI/CD gates.

## When not to use

Do not use this skill for generic LLM benchmarking, unrelated model leaderboards, or prompt-only rewrites that do not involve observed agent failures.
